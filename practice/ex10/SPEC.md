# ex10 — 階梯三 L5:API 分頁補抓(換域決賽)

**練習核心**:鏈、區塊、getLogs 全部消失。場景:你的後端要從第三方訂單系統
(Shopify 式 REST API)**補抓歷史訂單**。API 一次最多給一批,你要一批一批拉,
拉到沒有為止——而且程式隨時可能死,重啟後**從上次斷的地方繼續,一筆不漏不重**。

同一套「書籤與資料同進退」,全新的衣服。這題過了=階梯三完賽。

## 領域對照(自己完成翻譯,這就是 L5 的考點)

| 原本(indexer) | 這題(訂單補抓) |
|---|---|
| 鏈上事件 | 訂單 Order |
| getLogs(from, to) | `api.fetchAfter(sinceId, limit)`:「給我 **這筆 id 之後** 的訂單,最多 limit 筆」 |
| 書籤=區塊號(數字) | 書籤=**最後處理的訂單 id(字串)** |
| 鏈頭(掃到哪停) | **沒有鏈頭!** 停止條件=拿回來的批是**空的** |
| from = 書籤 + 1 | **不准做 +1!** 字串沒有算術——下一批的位置就是「上一批最後一筆的 id」 |

## 先看電影

```
庫是空的(cursor = null)
sync 開跑,batchSize = 3:
  fetchAfter(null, 3)        → [o1, o2, o3]  → commit([o1,o2,o3], "o3")
  fetchAfter("o3", 3)        → [o4, o5]      → commit([o4,o5], "o5")
  fetchAfter("o5", 3)        → []            → 收工,回傳 5(這次入庫筆數)

三天後有新訂單 o6, o7,再跑一次 sync:
  fetchAfter("o5", 3)        → [o6, o7]      → commit(...);回傳 2
                 ↑ 從庫裡的 cursor 接關,不是從頭

半途爆的版本:第 2 批 API 抽風 → 錯往上爆(這題不用接,單一資料源沒有別家要顧)
  → 但第 1 批已入庫、cursor="o3" → 修好再跑,從 o3 之後繼續,不漏不重
```

## 行為規則

1. `syncOrders(api, store, { batchSize })`:從庫的 cursor 接關(null=從頭),
   一批一批拉到空批為止,回傳**這次**新入庫的筆數。
2. 每批:訂單落庫+cursor 推進到**該批最後一筆的 id**,同一次 `commit` 完成。
3. 每次 fetchAfter 的 limit 都要傳 `batchSize`。
4. API 爆錯不用接——讓它往上爆;已 commit 的批必須完好(這你已經很熟了)。

## 不變量

1. cursor 永遠=最後成功入庫那批的最後一筆 id。
2. 重跑任意次,同一筆訂單只入庫一次(接關靠 cursor,不靠去重)。
3. 空批 → 不 commit、cursor 不動、回傳 0。

# ex7 — 階梯三 L2:換儲存底座(書籤搬進 JSON 檔)

**練習核心**:原版書籤存 Postgres,「事件+書籤同一筆交易」的原子性是資料庫送的。
這題把書籤搬到**本地 JSON 檔**——沒有資料庫了,原子性要你自己造。

## ⚠️ 這是核心 20%:crash 一致性

斷電可能發生在任何一行之間。規則:**不管斷在哪,重開機後讀到的書籤
要嘛是舊值、要嘛是新值,永遠不准是半截壞檔。**

## 你要對抗的物理事實(這就是考點)

| 操作 | 物理性質 |
|---|---|
| `writeFile`(寫檔) | **不是原子的**——斷電=檔案裡留半截內容 |
| `rename`(改名) | **是原子的**——要嘛整個發生、要嘛完全沒發生(OS 保證) |

業界標準解法叫 **write-temp-then-rename**:先把新內容寫進一個暫存檔,
寫完再把暫存檔改名成正式檔。你要自己把這個模式想通並實作出來——
想想:為什麼這樣斷電斷在任何一步,正式檔都完好?

## 行為規則(測試都會打)

1. 正式檔名固定叫 `checkpoint.json`,內容必須是合法 JSON(能被 `JSON.parse`)。
   暫存檔名你自己取,但**不准就是 `checkpoint.json`**。
2. `saveCheckpoint(disk, lastBlock)`:把書籤存進磁碟,斷電安全。
3. `loadCheckpoint(disk)`:回上次存的數字;從來沒存過 → 回 `null`。
4. 斷電模擬:測試會用 `disk.armCrash(...)` 在你的某個操作上裝炸彈,
   炸到會 throw `CrashError`(寫檔的炸彈還會留半截內容)。你的 save 被炸=讓它爆,
   不用 try/catch——重點是爆完之後 `loadCheckpoint` 還能讀到完好的舊值。
5. 斷電後復活:殘留的暫存檔不准擋住下一次 save。
6. **陷阱**:有一顆測試在 `write:checkpoint.json` 上裝炸彈——直接寫正式檔的人
   會當場引爆+留半截 JSON;走對路的人根本不會碰到它。

## 必須維持的不變量

1. 任何時刻(含斷電後)`checkpoint.json` 若存在,內容必是合法 JSON。
2. 斷電後 load 到的值 ∈ {舊值, 新值},不存在第三種。
3. save 成功返回 ⇒ load 一定拿到新值。

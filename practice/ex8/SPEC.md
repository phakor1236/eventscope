# ex8 — 階梯三 L3:一掃多約(參與者變多)

**練習核心**:原版 indexer 只追一個合約(PriceDuel),書籤一張就夠。
這題:**同一個掃描迴圈同時追 3 個合約**,各自 deploy block 不同、各自有書籤,
而且——**一家的 RPC 失敗,不准拖累另外兩家的書籤前進**。

## ⚠️ 這是核心 20%:狀態機 × N + 故障隔離

真實世界的 indexer 都長這樣:一個 daemon 追一串合約,其中一個合約的 RPC
偶爾抽風是日常。抽風的那家停在原地等下一輪,其他家照常前進。

## 與 ex6 的差異

| ex6 | 這題 |
|---|---|
| 1 個合約,書籤 1 張 | 3 個合約,**書籤各一張,互不相干** |
| confirmations 緩衝帶 | 拿掉(考點不在這),掃到鏈頭 |
| store 書籤初始值=deployBlock−1(store 幫你算好) | store 沒掃過回 **null**——「null 該從哪開始掃」是你的事,而且**每家的起點不一樣** |
| getLogs 不會失敗 | `getLogs` 可能 **throw RpcError**(測試會裝故障) |
| 不回東西 | 回戰報 `{ ok: [...], failed: [...] }` |

## 行為規則(測試都會打)

1. `catchUpAll(chain, store, contracts, { chunkSize })`:對清單裡每家合約,
   從**它自己的書籤**(沒掃過=從**它自己的 deployBlock**)掃到鏈頭,分段規則同 ex6。
2. `getLogs(合約, from, to)`:問鏈要**這一家**的事件。每家的事件各歸各家,
   不准串門子。
3. 某家 getLogs 爆 `RpcError` → **接住它**:這家記進 `failed`,書籤留在
   「最後一次成功 commit 的段尾」;**剩下的合約照常處理,函數本身不准爆**。
4. 半途爆:一家掃了 3 段、第 2 段爆 → 第 1 段已 commit 的成果保留,
   書籤=第 1 段尾。下一輪故障排除後,從那裡續掃,一筆不漏。
5. 全部成功 → `failed` 空陣列;空合約清單 → `{ ok: [], failed: [] }`,不碰鏈。

## 必須維持的不變量

1. 每家書籤獨立:A 的進度永不影響 B 的書籤。
2. 書籤只反映**成功 commit** 的段:爆掉那段之前的保留、之後的沒有。
3. 一家失敗,函數不 throw、其他家照常推進。
4. 每段涵蓋區塊數 ≤ chunkSize,段與段首尾相接(每家各自算)。

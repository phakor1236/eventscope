# ex2 — 階梯二 L1:帳目明細化(insert 回傳兩本帳)

**練習核心**:同 ex1 的冪等寫入,但**回傳值的業務規則變了**——呼叫方不再只想知道
「新增幾筆」,還要知道「被擋下幾筆」。真實動機:indexer 的監控面板要能看出
「這輪掃描重複率多高」,重複率突然飆高往往代表書籤壞了在重掃。

## 與 ex1 的差異

| | ex1 | 這題 |
|---|---|---|
| `insert` 回傳 | `number`(新增筆數) | `InsertReport = { added, skipped }` |
| 「跳過」 | 不關心,直接無視 | **要精確計數**,含同批內重複 |
| 其他行為 | — | first-write-wins、鍵設計、`has` 全部照舊 |

## 行為規則(測試都會打)

1. `insert(batch)` 回傳 `{ added, skipped }`:
   - `added` = 這次真正存進庫的筆數(同 ex1 的回傳值)
   - `skipped` = 這次被冪等機制擋下的筆數(跨批重複、同批內重複都算)
2. 每一筆進來的事件,**恰好被記進其中一本帳**:不是 added 就是 skipped,
   不能兩本都記、也不能兩本都漏。
3. first-write-wins 照舊:被 skip 的筆,內容再新也整筆丟棄。
4. 鍵照舊是兩零件 `(tx_hash, log_index)`,黏法的分隔符陷阱照舊。
5. 空批 → `{ added: 0, skipped: 0 }`,庫不動。

## 必須維持的不變量

1. 任何一次呼叫:`added + skipped === batch.length`。
2. 歷次 `added` 加總 === `count()`(每筆入庫事件恰好被 added 記過一次)。
3. 庫裡的筆數 = 出現過的不同 `(tx_hash, log_index)` 數(繼承 ex1)。
4. 已存資料永不被改寫(繼承 ex1)。
